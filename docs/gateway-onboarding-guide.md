# mall-gateway 从零上手指南（先规划，后实现）

## 1. 这份文档解决什么问题

如果你之前没做过网关，这份文档的目标是：

- 帮你先建立正确的网关认知，不一上来就堆配置。
- 给你一条可以直接执行的落地路径（按阶段推进）。
- 每个阶段都给出完成标准，避免“看起来配了很多，实际上不可用”。

本指南基于当前项目结构：

- mall-gateway：网关服务
- mall-auth：认证服务
- mall-user、mall-product、mall-order：业务服务

---

## 2. 先理解：网关到底做什么

网关是系统对外的统一入口，核心价值是“统一治理”，而不是承载业务逻辑。

网关适合做：

- 请求路由转发
- 统一认证前置校验（如 token 是否有效）
- CORS、限流、黑名单、审计日志
- 统一错误出口（例如 401、403、429）

网关不适合做：

- 复杂业务逻辑（如下单规则、库存扣减）
- 细粒度业务授权判定（通常在业务服务内部）
- 用户状态长期存储（网关尽量无状态）

---

## 3. 开始前准备（30 分钟）

### 3.1 先确认依赖关系和调用方向

你要先明确这一条主链路：

客户端 -> mall-gateway -> 各业务服务

原则：

- 对外只暴露网关端口。
- 业务服务尽量不直接暴露给公网。

### 3.2 先画一版最小路由表

建议先列一个表（可写在你自己的笔记或任务单里），字段如下：

- 路由 ID
- 上游路径（客户端访问）
- 下游服务（转发目标）
- 是否鉴权
- 限流等级
- 是否白名单

你现在已有的基础路由可以作为初始版本：

- auth
- user
- product
- order

### 3.3 先统一安全语义

在团队内先约定：

- 401：未登录或 token 无效
- 403：已登录但无权限
- 429：触发限流

这个约定必须在网关和各服务中保持一致。

## 4. 分阶段实施路径（建议按顺序）

## 阶段 A：先只做“可路由”

目标：请求能稳定从网关转发到目标服务。

你要做的事：

- 确认每个业务域的路径前缀（auth、user、product、order）。
- 明确是否要保留前缀传给下游，还是在网关剥离前缀。
- 为每个路由定义最小可用规则（路径匹配 + 转发）。

完成标准：

- 4 个核心域都能经过网关访问。
- 不经过网关的直连路径不作为主入口。

常见坑：

- 路径前缀设计反复变更，导致前后端频繁改接口。
- 下游服务路径与网关重写策略不一致。

## 阶段 B：接入认证前置校验

目标：把“未认证请求”尽早拦在网关。

你要做的事：

- 定义白名单接口（登录、注册、健康检查等）。
- 定义非白名单接口的 token 校验策略。
- 定义校验通过后，向下游透传哪些用户上下文信息。

完成标准：

- 白名单请求可通行。
- 非白名单请求无 token 或 token 非法时返回统一 401。
- 下游服务可获取用户基本身份信息（例如用户 ID）。

常见坑：

- 白名单匹配过宽，导致敏感接口被误放行。
- 网关和下游重复做冲突校验，出现不一致结果。

## 阶段 C：统一跨域与错误出口

目标：让前端调用行为可预期，错误返回可治理。

你要做的事：

- 固化 CORS 规则（来源、方法、请求头、凭证策略）。
- 统一网关返回结构（至少统一错误码和错误信息格式）。
- 区分网关错误与下游业务错误。

完成标准：

- 前端在开发/测试环境跨域行为一致。
- 401/403/429/5xx 响应语义稳定。

常见坑：

- 使用允许所有来源同时允许凭证，导致浏览器策略冲突。
- 错误格式不统一，前端兼容成本高。

## 阶段 D：加限流与基础防护

目标：避免网关成为“放大器”，先挡住明显异常流量。

你要做的事：

- 先按接口风险分级：登录接口严格，查询接口相对宽松。
- 确定限流维度（IP、用户、接口）。
- 设定触发限流后的统一响应（429）。

完成标准：

- 高频请求命中限流可见、可观测。
- 限流策略不会误伤大多数正常请求。

常见坑：

- 只按 IP 限流，NAT 场景误伤较大。
- 阈值拍脑袋设置，未经过压测验证。

## 阶段 E：补齐日志与可观测性

目标：问题发生后可以快速定位，而不是“猜”。

你要做的事：

- 记录基础访问日志：方法、路径、状态码、耗时、客户端标识。
- 透传并统一 traceId。
- 敏感字段脱敏（如 token、手机号）。

完成标准：

- 一次异常请求可以从网关追到下游。
- 安全审计能定位到关键失败场景（登录失败、鉴权失败、限流触发）。

常见坑：

- 日志过多但没有结构化字段，无法检索。
- 明文打印敏感头信息，带来泄露风险。

## 5. 你的当前项目可以怎么起步（建议一周节奏）

Day 1：认知与边界

- 写出网关职责清单与非职责清单。
- 和现有安全职责文档对齐语义。

Day 2：路由收敛

- 固化 4 个域的路径前缀规则。
- 明确前缀保留/剥离策略。

Day 3：认证主链路

- 输出白名单清单。
- 定义 token 校验失败行为和透传字段。

Day 4：跨域与错误出口

- 统一 CORS 规则。
- 固化 401/403/429/5xx 返回规范。

Day 5：限流与观测

- 配置分级限流阈值。
- 对关键路径补齐日志与追踪字段。

Day 6：联调

- 与 auth、user、product、order 做端到端联调。
- 覆盖正常与异常路径。

Day 7：压测与上线准备

- 做小规模压测，校准限流阈值。
- 准备回滚策略与监控告警阈值。

## 7. 常见误区（新手高频）

- 一上来先写大量过滤器，结果边界不清，后续返工。
- 把授权逻辑全部堆在网关，导致网关过重。
- 路由命名和路径规范不统一，维护成本高。
- 没有统一错误语义，前端出现大量 if-else 分支。
- 没做压测就上线限流策略，造成误封或失效。

---

## 8. 下一步建议

当你准备开始真正配置时，下一份文档建议写成：

- 网关配置基线清单（环境、路由、过滤器顺序、日志级别）
- 白名单与鉴权策略清单（按接口分类）
- 限流参数说明（每类接口为什么这么配）

这样你后续写实现时，会非常稳，不容易走弯路。
